# 眼动数据用于认知障碍检测 — 调研报告

> 调研日期：2026-06-12
> 目的：评估 Huatuo 项目中眼动数据的可用性，设计 EEG + 眼动多模态融合方案

## 一、Huatuo 项目眼动数据现状

### 1.1 数据库字段

数据库表 `eeg_data` 中有 5 个眼动相关字段：

| 字段 | 类型 | 含义 |
|------|------|------|
| `blink_l` | DOUBLE PRECISION | 左眼眨眼检测 (0 或 1) |
| `blink_r` | DOUBLE PRECISION | 右眼眨眼检测 (0 或 1) |
| `gaze_x` | DOUBLE PRECISION | 注视方向向量 X 分量 / 位置坐标 X |
| `gaze_y` | DOUBLE PRECISION | 注视方向向量 Y 分量 / 位置坐标 Y |
| `gaze_z` | DOUBLE PRECISION | 注视方向向量 Z 分量 / 位置坐标 Z |

### 1.2 值域统计（排除全零数据后）

| 通道 | 最小值 | 最大值 | 均值 |
|------|--------|--------|------|
| gaze_x | -2.38 | 2.15 | 0.29 |
| gaze_y | -1.92 | 1.05 | -0.75 |
| gaze_z | -5.63 | -1.55 | -3.92 |
| blink_l/r | 0 | 1 | — |

`gaze_z` 始终为负值，符合 VR 头盔中注视方向朝前方的预期。

### 1.3 采集方式

项目中有两个客户端文件实现了眼动采集，机制不同：

#### DataControl.cs（较早版本）

```csharp
var pose = eyeTrackingData.eyeDatas[2].pose;
var rotation = new Quaternion(-pose.position.x, -pose.position.y, pose.position.z, pose.orientation.x);
Vector3 dir = rotation * Vector3.forward;
currentEyeData = new float[3] { dir.x, dir.y, dir.z };
```

- `gaze_x/y/z` 是通过四元数旋转 `Vector3.forward` 得到的**单位方向向量**
- `blink_l/r` 来自 `GetEyeOpenness()`，理论上是 0.0-1.0 的连续值

#### EyeNeuroManagerService.cs（较新版本）

```csharp
var pose = eyeTrackingData.eyeDatas[2].pose;
eyePoints.Add(CreatePoint(GazeXChannel, pose.position.x, sampleTime));
// ...
int blinkStatus = PXR_MotionTracking.GetEyeBlink(ref blinkTimestamp, ref isLeftBlink, ref isRightBlink);
```

- `gaze_x/y/z` 直接使用 `pose.position`（**位置坐标**，非单位向量）
- `blink_l/r` 使用 `GetEyeBlink()` API，返回严格的布尔值 (0/1)

**关键区别**：两种实现的 `gaze_x/y/z` 语义不同（方向向量 vs 位置坐标），但数据库中无法区分来源。

### 1.4 采样率

| 信号类型 | 采样率 | 采集位置 |
|----------|--------|---------|
| EEG | 250Hz (4ms) | `BCI_onRawDataEvent` 回调 |
| 眼动 | ~62.5Hz (16ms) | Unity `FixedUpdate()` |

眼动数据量约为 EEG 的 **1/4**。这是因为眼动在 Unity 物理更新循环中采集，而非来自 NeuroXess SDK。

### 1.5 数据覆盖率

| 指标 | 数值 |
|------|------|
| 总用户数 | 90 |
| 有眼动数据的用户 | 29 (32%) |
| 无眼动数据的用户 | 61 (68%) |
| 眼动数据行占比 | 14-18% |

从 2026-05-26 之后的用户**全部没有眼动数据**，可能是因为后续采集时未启用眼动追踪或使用了不支持眼动的设备。

### 1.6 存储结构

眼动和 EEG 在数据库中是**分开存储的行**，通过时间戳关联：

| 数据类型 | 行数 (user 5201704) |
|----------|---------------------|
| 只有 EEG | 28,075 |
| 只有 Gaze | 6,225 |
| 既无 EEG 也无 Gaze | 9,618 |

## 二、与认知障碍相关的眼动特征

### 2.1 扫视（Saccade）特征

扫视是快速的眼球运动，用于将注视点从一个目标转移到另一个目标。

- **扫视延迟（Saccade Latency）**：从刺激呈现到扫视开始的时间。AD/MCI 患者在反扫视任务中的反应时间显著延长，比健康对照组增加约 30-50ms。这是最具区分力的特征之一。
- **扫视速度（Saccade Peak Velocity）**：AD 患者的峰值扫视速度降低，反映眼动控制系统退化。
- **扫视幅度（Saccade Amplitude）**：认知障碍患者的扫视幅度趋于减小，视觉探索范围缩窄。
- **反扫视错误率（Anti-saccade Error Rate）**：研究最充分的指标。AD 患者的错误率可达 40-80%，健康老年人通常在 15-25%，MCI 患者介于两者之间。该指标与前额叶执行功能高度相关。

### 2.2 注视（Fixation）特征

注视是眼球相对稳定、聚焦于某一目标的状态。

- **注视时长（Fixation Duration）**：认知障碍患者的平均注视时间延长，反映信息处理速度下降。健康人约 200-300ms，AD 患者可达 400-600ms。
- **注视点数量（Fixation Count）**：在自由观看任务中，认知障碍患者的总注视点数量减少。
- **注视空间分布（Fixation Spatial Distribution）**：AD 患者的注视点更集中于图像中心（中心偏差增强），视觉探索的分散度降低。
- **注视稳定性（Fixation Stability）**：MCI/AD 患者在维持稳定注视时表现出更多微扫视（microsaccade）异常。

### 2.3 瞳孔（Pupil）特征

- **瞳孔直径变化（Pupil Diameter Change）**：在认知负荷任务中，MCI/AD 患者的瞳孔扩张反应减弱。与蓝斑核（Locus Coeruleus）去甲肾上腺素系统退化有关。
- **瞳孔光反射（Pupillary Light Reflex, PLR）**：AD 患者的瞳孔光反射潜伏期延长、收缩幅度减小。
- **任务诱发瞳孔反应（Task-Evoked Pupillary Response）**：被认为是最有前景的早期筛查指标之一，可在认知测试期间实时反映认知负荷。

> 注：Huatuo 项目当前未采集瞳孔直径数据。

### 2.4 眨眼（Blink）特征

- **眨眼频率（Blink Rate）**：与多巴胺系统功能相关。AD 患者的眨眼频率可能异常（多数研究报告增加）。
- **眨眼时长（Blink Duration）**：认知障碍患者眨眼持续时间可能延长。
- **自发眨眼与任务相关眨眼的比值**：反映注意力分配能力。

### 2.5 平滑追踪（Smooth Pursuit）

- **追踪增益（Pursuit Gain）**：AD 患者平滑追踪的增益降低（实际眼动速度/目标速度），出现更多追赶性扫视。
- **追踪延迟**：平滑追踪反应的延迟增加。

### 2.6 视觉探索模式（Visual Exploration）

- **扫描路径长度（Scanpath Length）**：观看复杂场景时，认知障碍患者的扫描路径显著缩短。
- **熵值（Entropy）**：注视序列的信息熵降低，反映探索模式变得刻板和可预测。
- **人脸注视模式**：AD 患者在观看人脸时表现出异常的注视分配（减少对眼睛区域的注视、增加对嘴部区域的注视）。

## 三、Huatuo 可提取的眼动特征

基于当前采集的 5 个通道（blink_l/r, gaze_x/y/z），可提取以下特征：

### 3.1 眨眼特征（来自 blink_l/r）

| 特征 | 计算方法 | 临床意义 |
|------|---------|---------|
| blink_rate | 眨眼次数 / 总时长 | AD 患者眨眼频率升高 |
| blink_duration_mean | 连续 1 的平均持续时长 | 认知障碍者眨眼时长延长 |
| blink_interval_mean | 两次眨眼间隔的均值 | 反映注意力分配 |
| blink_interval_std | 两次眨眼间隔的标准差 | 眨眼节律稳定性 |
| blink_asymmetry | \|blink_l_rate - blink_r_rate\| | 左右不对称可能反映单侧脑功能退化 |

### 3.2 注视稳定性特征（来自 gaze_x/y）

| 特征 | 计算方法 | 临床意义 |
|------|---------|---------|
| gaze_x_mean, gaze_y_mean | 均值 | 注视中心位置 |
| gaze_x_std, gaze_y_std | 标准差 | 注视分散度（越小越集中于中心，AD 特征） |
| gaze_x_cv, gaze_y_cv | 变异系数 = std/mean | 注视变异度 |
| gaze_dispersion | std_x * std_y | 注视空间覆盖面积代理 |
| gaze_path_length | 相邻点欧氏距离之和 | 扫描路径总长度（AD 患者缩短） |

### 3.3 扫视特征（来自 gaze_x/y 的一阶差分）

| 特征 | 计算方法 | 临床意义 |
|------|---------|---------|
| saccade_count | 速度超过阈值的次数 | 扫视频次 |
| saccade_amplitude_mean | 扫视幅度均值 | AD 患者扫视幅度减小 |
| saccade_velocity_mean | 扫视速度均值 | AD 患者峰值速度降低 |
| saccade_velocity_max | 扫视速度最大值 | 眼动控制系统退化指标 |

> 注：62.5Hz 采样率对高速扫视的捕捉精度有限，扫视速度特征的可靠性会受影响。

### 3.4 缺失指示特征

| 特征 | 计算方法 | 用途 |
|------|---------|------|
| has_eye_tracking | 是否有眼动数据 (0/1) | 区分有无眼动数据的用户 |
| gaze_valid_ratio | 有效数据点 / 总数据点 | 数据质量评估 |
| blink_valid_ratio | 有效眨眼数据比例 | 数据质量评估 |

## 四、特征工程 vs. 端到端深度学习

### 4.1 传统特征工程方法（主流，约 70% 的研究）

- **流程**：原始眼动数据 → 事件检测（I-VT 或 I-DT 算法）→ 提取统计特征 → 传统 ML 分类器
- **常用统计量**：均值、标准差、中位数、四分位距、偏度、峰度、变异系数
- **分类器**：Random Forest、SVM、XGBoost、Logistic Regression
- **优势**：可解释性强、样本需求量小（50-200 人即可训练）、临床可接受度高
- **典型精度**：MCI vs. 健康对照的分类 AUC 约 0.75-0.90

代表性研究：
- Lagun et al. (2011)：反扫视任务特征 + SVM，区分 MCI 与健康对照的准确率约 79%
- Yang et al. (2023)：多任务眼动特征 + XGBoost，AUC 达到 0.88
- Zagermann et al. (2016)：自由观看任务中的注视模式 + RF 进行认知状态分类

### 4.2 端到端深度学习方法（新兴，约 20-30% 的研究）

- **流程**：原始/低层处理的眼动时序数据 → 深度学习模型 → 分类结果
- **常用架构**：1D-CNN、LSTM/GRU、CNN-LSTM、Transformer、TCN
- **优势**：自动学习特征、可能发现人类未注意到的模式
- **劣势**：需要大量数据（通常 >500 人）、可解释性差、容易过拟合
- **典型精度**：与传统方法相当或略优，但在小数据集上往往不如特征工程方法

### 4.3 混合方法（推荐用于 Huatuo）

- **深度学习辅助特征提取**：使用预训练模型提取 embedding，再用传统分类器
- **注意力机制 + 手工特征**：将手工特征作为先验知识注入深度学习模型
- **数据增强 + 迁移学习**：通过数据增强弥补样本不足

> Huatuo 项目当前只有 ~90 个用户，其中 29 个有眼动数据，建议以**特征工程方法**为主。

## 五、眼动 + EEG 多模态融合方法

### 5.1 为什么融合？

- **EEG**：捕获神经振荡活动（Theta/Alpha/Beta 功率、ERPs、功能连接），反映大脑内部处理过程
- **眼动**：捕获外部行为表现（注视、扫视、眨眼反应），反映注意力分配和认知输出
- 两者提供互补信息：EEG 是"内部视角"，眼动是"外部视角"

### 5.2 融合策略

#### 早期融合（Feature-level Fusion）

将 EEG 特征和眼动特征拼接为一个特征向量，输入到单一分类器中。

- 优势：简单直接，保留所有特征间交互
- 劣势：特征维度高，可能引入噪声
- **适用性**：最适合 Huatuo 项目当前阶段

#### 晚期融合（Decision-level Fusion）

分别训练 EEG 模型和眼动模型，在决策层进行融合（加权平均、投票、堆叠）。

- 优势：各模态独立优化，灵活性高
- 劣势：忽略了模态间的底层交互

#### 中间融合（Hybrid Fusion）

使用深度学习架构学习跨模态共享表示：

- **Cross-Modal Attention**：用 EEG 特征作为 Query，眼动特征作为 Key/Value
- **Multimodal Autoencoder**：将两种模态编码到共享潜在空间
- **Transformer-based Fusion**：将两种模态的 token 序列拼接，通过自注意力学习交互

### 5.3 时间对齐策略

| 方法 | 说明 | Huatuo 适用性 |
|------|------|--------------|
| 硬件同步 | 使用统一时钟源（如 LSL） | 需要改造 Unity 客户端 |
| 时间戳对齐 | 基于 UNIX 时间戳插值 | 当前方案，已实现 |
| 事件标记对齐 | 通过共同事件标记对齐 | 可用于游戏任务期间 |
| 动态时间规整 (DTW) | 处理时间漂移 | 离线分析可用 |

### 5.4 代表性研究

- Mei et al. (2023)：EEG + 眼动融合用于 MCI 检测，注意力机制融合，AUC 比单模态提升约 5-10%
- Hajra et al. (2023)：EEG + 瞳孔反应融合用于阿尔茨海默病早期筛查
- Kolkhorst et al. (2024)：VR 环境中 EEG + 眼动数据进行认知评估

## 六、缺失数据处理策略

### 6.1 缺失原因分类

- **眨眼相关丢失**：生理性的，持续约 100-400ms，可预测
- **追踪丢失**：眼睛偏离追踪范围、镜片反光、头部运动等
- **设备故障**：硬件/软件问题导致的系统性丢失
- **用户级缺失**：部分用户完全没有眼动数据（Huatuo 中占 68%）

### 6.2 短间隙插值（<100ms）

- 线性插值：最简单，适用于平稳信号段
- 三次样条插值：更平滑，适用于瞳孔数据
- Huatuo 项目已有类似实现：`receiver_sender.py` 中的 `interpolate_missing_values()`

### 6.3 长间隙处理（>500ms）

- 通常排除该段数据，不进行插值
- 在滑动窗口特征提取中跳过包含大量缺失的窗口
- Huatuo 项目已有类似机制：`feature_extraction.py` 中的窗口质量检查

### 6.4 用户级缺失处理策略

| 策略 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| 丢弃无眼动用户 | 只用 29 个有眼动的用户 | 数据一致性好 | 样本量大幅减少 (90→29) |
| 填零 + 缺失标志 | 无眼动用户的眼动特征填 0，增加 has_eye_tracking 标志 | 保留所有样本 | 引入噪声 |
| 分别建模 | 有眼动用多模态，无眼动用纯 EEG | 各自优化 | 需要维护两个模型 |
| 多重插补 | 基于 EEG 特征预测眼动特征 | 保留样本且不简单填零 | 实现复杂，可能引入偏差 |

**推荐策略**：填零 + 缺失标志（适用于传统 ML 模型，RF/XGB 能自动学习哪些特征有用）

## 七、事件检测算法

### 7.1 I-VT（Velocity-Threshold Identification）

基于速度阈值区分注视和扫视：
- 计算相邻采样点间的角速度
- 速度 < 30°/s → 注视
- 速度 > 100°/s → 扫视
- 中间区域 → 可能是微扫视或噪声

### 7.2 I-DT（Dispersion-Threshold Identification）

基于空间散布度识别注视：
- 计算窗口内采样点的空间散布度（max_x - min_x + max_y - min_y）
- 散布度 < 1-2° 视角 → 注视
- 否则 → 扫视

### 7.3 Microsaccade 检测（Engbert & Kliegl 算法）

- 基于速度阈值的椭圆形检测
- 速度超过中位速度 6 倍的采样点标记为微扫视
- 微扫视与注意力转移和认知负荷相关

## 八、数据预处理建议

### 8.1 噪声滤波

- **注视坐标数据**：中值滤波去除跳变噪声，低通滤波（4-8Hz）平滑轨迹
- **眨眼数据**：无需滤波，直接使用 0/1 值

### 8.2 缺失值处理

- 眨眼导致的数据丢失（~100-400ms）：线性插值
- 追踪丢失（>500ms）：标记并排除该段
- 有效性阈值：有效追踪比例 >70% 的数据才纳入分析

### 8.3 坐标归一化

- 将 gaze_x/y 归一化到 [0,1] 或 [-1,1] 范围
- 对于 VR 场景：可转换为视角坐标（degree of visual angle）
- 减去每个用户的均值，消除个体间基线差异

## 九、对 Huatuo 项目的建议总结

| 方面 | 当前状态 | 建议改进 |
|------|----------|---------|
| 眼动数据采集 | 已有 blink_l/r, gaze_x/y/z | 统一客户端采集方式，确认采样率 |
| 特征工程 | 仅使用 EEG 特征 | 添加眨眼频率、注视稳定性等眼动特征 |
| 数据预处理 | 有基础插值 | 针对眼动特性设计专门预处理流程 |
| 多模态融合 | 仅 EEG + 游戏行为 | 将眼动特征纳入特征向量（早期融合） |
| 缺失处理 | 有基础插值 | 增加通道级有效率评估和缺失指示特征 |
| 诊断模型 | RF/XGB | 保持现有模型，眼动特征作为额外输入 |

### 推荐的实施优先级

1. **眨眼特征**（最稳健）：blink_rate, blink_interval_mean, blink_interval_std
2. **注视稳定性特征**：gaze_x/y 的标准差、变异系数
3. **扫视代理特征**：gaze_x/y 一阶差分的统计量
4. **缺失指示特征**：has_eye_tracking, gaze_valid_ratio
